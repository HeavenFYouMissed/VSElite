## The architecture (why it's weird)

V3Code = a fork of Void = a fork of VS Code OSS **1.99.3**, vendored (the whole VS Code
source is copied into this repo). The AI chat is a **React app** that Void bolted on top of
VS Code's non-React core. That bolt-on is why the build has extra steps.

### The build pipeline (4 stages)

```
src/...react/src/   --(scope-tailwind)-->  react/src2/   --(tsup)-->  react/out/   --(gulp compile-client OR dev.ps1 copy)-->  out/vs/.../react/out/   --(V3Code.exe loads)
   you edit here        prefixes classes        bundles            React bundle           host bundle the app actually loads
```

- **`src/`** — you edit here (`.tsx` React + `.ts` services).
- **`react/src2/`** — GENERATED by scope-tailwind (prefixes Tailwind classes `void-`). Never hand-edit.
- **`react/out/`** — tsup bundles src2 here.
- **`out/`** — the VS Code host output. `gulp compile-client` compiles all `.ts` here AND copies `react/out` in.
- The running app loads from **`out/`**.

### Two kinds of edits → two build paths
- **React/CSS edit** (`react/src/**/*.tsx|css`): fast path — scope-tailwind + tsup + **copy** to host. `Ctrl+R` to see it. ~15s.
- **Service edit** (`.ts` in `browser/` or `common/`): needs **full `gulp compile-client`** (~3 min), because gulp compiles those into `out/`.

## BLACK SCREEN — the #1 problem and its causes

A black window with only minimize/maximize/close = the **renderer crashed before drawing**.
There are TWO root causes, both now handled by `dev.ps1`/`build.ps1`:

### Cause 1 (the big one): missing `VSCODE_DEV=1`
A `compile-client` build is a **DEV build**. It must be launched with these env vars (this is
what `scripts/code.bat` does):
```powershell
$env:VSCODE_DEV = "1"        # makes the window load workbench-dev.html + dev module loader
$env:NODE_ENV   = "development"
$env:VSCODE_CLI = "1"
```
Without `VSCODE_DEV=1`, the app runs in **production** mode, expects bundled output a dev
build doesn't have, and `window.MonacoBootstrapWindow` is undefined → `workbench.js` crashes
calling `.load()` on undefined → black screen.

### Cause 2: missing `out/nls.messages.json`
`gulp compile-client` runs `rimraf('out')` then compiles with `build=false`, which does NOT
run the NLS plugin — so `out/nls.messages.json` is never generated. But `src/main.ts` requires
it at boot. Missing → ENOENT → undefined NLS → crash. Fix: write `[]` to `out/nls.messages.json`
(dev uses inline English, so empty is fine). `dev.ps1`/`build.ps1` do this automatically.

### How to diagnose a black screen
```powershell
# Launch with logging and read the renderer console:
$env:VSCODE_DEV="1"; $env:NODE_ENV="development"; $env:VSCODE_CLI="1"
.build\electron\V3Code.exe . --user-data-dir .tmp\user-data --enable-logging --log=trace 2> err.txt
# then read err.txt — look for "CONSOLE" lines and TypeErrors.

# Healthy app = ~8 V3Code processes + a renderer.log created in .tmp/user-data/logs/<session>/window1/
Get-Process -Name V3Code | Measure-Object   # 5-8 = good; 1-2 = crashed
```

## "My changes don't show up"
1. Did you edit a `.tsx`? Run `.\dev.ps1` (or it's already watching) → **press Ctrl+R in V3Code**. A reload is required; the bundle is loaded once at startup.
2. Did you edit a `.ts` service? You need `.\dev.ps1 -FullGulp` (fast copy won't recompile services).
3. Still nothing? Confirm the host bundle updated:
   ```powershell
   # the string you changed should appear here:
   Select-String "your-text" out\vs\workbench\contrib\void\browser\react\out\sidebar-tsx\index.js
   ```

## Don't run gulp while V3Code is open
`gulp compile-client` does `rimraf('out')`. If V3Code is running it holds `out/` files locked,
producing a half-written `out/` → black screen. `dev.ps1` kills V3Code before building.

## Known-good run checklist
- [ ] Build succeeded (`tsup` exit 0 / `gulp` "0 errors").
- [ ] `out/nls.messages.json` exists.
- [ ] Launched with `VSCODE_DEV=1` (via `dev.ps1`/`build.ps1`/`code.bat`).
- [ ] No other V3Code instance was running during a gulp build.
- [ ] ~5-8 V3Code processes alive after launch.
