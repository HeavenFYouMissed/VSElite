## Architecture

V3Code = Void Editor + Context Bridge (built-in, not MCP) + local embeddings (planned).

```

│   │   ├── contextBridge/                  ← VENDORED bridge code (Phase B)
│   │   │   ├── lsp-bridge.ts               ← Spawns tsserver, JSON-RPC
│   │   │   ├── memory.ts                   ← .v3code/notes.json
│   │   │   └── types.ts
│   │   └── mcpService.ts                   ← For user's own external MCP servers (untouched)

```

**Key principle:** Context Bridge ships as native built-in tools. NOT as an MCP server. NOT as an external process. No hardcoded paths. The LSP bridge code lives inside the editor binary. Always on. Always works.

## The 9 Tools V3Code ships

All native built-ins via `toolsService.ts`. None exposed over MCP.

| Tool | Purpose |
|---|---|
| `get_symbol_context` | Symbol neighborhood — def, callers, callees, refs, type hierarchy, diagnostics, attached notes |
| `get_file_context` | File structural overview — symbols + imports + diagnostics |
| `get_call_graph` | Recursive N-level caller/callee tree (depth 1-4) |
| `get_file_dependencies` | Direct imports + reverse dependents |
| `find_text` | Grep with context (LSP-aware ranking later) |
| `pack_context` | Task-typed token-budgeted bundle for a symbol |
| `remember` | Attach persistent note to symbol — survives across sessions |
| `forget` | Remove note |
| `list_notes` | List notes in workspace |

Notes persist in `<workspace>/.v3code/notes.json` (V3Code brand) — was `.context-bridge/notes.json` in the standalone MCP server.

### 2026-05-27 — Workbench boot fix: semanticIndex layer violation stubbed
- **Symptom**: compile was green and `out/main.js` existed, but launching V3Code opened a near-empty dark window. DevTools showed `bootstrap-window.ts:151 [uncaught exception]: TypeError: Failed to resolve module specifier "path"` followed by `workbench.ts:312`. Window never reached the agent panel.
- **Root cause**: the entire `src/vs/workbench/contrib/void/common/semanticIndex/` stack uses Node builtins (`path`, `fs`, `os`, `crypto`, native `@vscode/sqlite3`) but was wired into the renderer through `browser/void.contribution.ts` (line 74: `import '../common/semanticIndex/semanticIndexService.js'`). The renderer's native ESM loader refuses bare specifiers — TypeScript's compile-time `import { join } from 'path'` ships as a literal `import "path"` into the browser bundle and explodes at module-eval time.
- **Layer rule for V3Code**: code under `common/` MUST be browser+node safe. Anything touching Node builtins or native modules belongs in `node/` (node-process service), `electron-browser/` (renderer-with-node), or behind an IPC boundary. `import type { Database } from '@vscode/sqlite3'` is fine because it's type-only and erased; `import { join } from 'path'` is fatal.
- **Workaround shipped**: created `browser/semanticIndexBrowserStub.ts` — implements `ISemanticIndexService` as a no-op (status stays `uninitialized`, `retrieve()` returns `[]`, `rebuild()` does nothing). Swapped the contribution import to load the stub instead. `semanticIndexConfiguration`, `semanticIndexActions`, `semanticIndexStatusBar` still register and resolve `ISemanticIndexService` cleanly — they just get the stub. Compile re-passed in 2.2 min; workbench should boot.
- **Real fix (deferred)**: move `SemanticIndexService` into `electron-main/` or `node/` and front it with IPC channels; keep the types decorator in `common/`. Until then the stub is the seam.
- **Suggested improvement**: add a CI lint that fails if any file under `**/common/**` contains a static `import ... from 'path'|'fs'|'os'|'crypto'|'child_process'|'worker_threads'|'http'|'https'|'net'|'tls'|'stream'|'zlib'|'@vscode/sqlite3'|'node-llama-cpp'`. Would have caught this at PR time instead of launch time. Ask Daniel: yes/no on adding it to the hygiene gulp task?

### 2026-05-27 — Cursor-agent audit triage: agent panel hardening + indexer commercial polish
- **Triage approach**: another agent (Cursor) handed off a 16-item audit (6 agent-panel + 10 indexer). Took it as triage input, not a checklist. Shipped 12 (the genuinely valid ones); skipped 4 with explicit rationale (see Recent Changes). User mandate was "stop asking, use your judgement, the way it should be done for a commercial platform" — executed in one pass without further check-ins.
- **The one real bug in the audit**: agent panel was calling `findEditors(VoidChatEditorInput.RESOURCE)` + `closeEditors([...all])` on exit. If the user manually opened a chat editor in a third tab group, leaving agent mode would close it. Now stash `_ownedEditor: IEditorIdentifier = { editor: pane.input!, groupId: pane.group.id }` from `openEditor`'s returned pane and only close that specific identifier. Pattern worth remembering: **services that open editors must track what they opened, not search by resource and close all matches**.
- **Audit value calibration**: of the 16 items, ~75% landed (4 panel + 8 indexer = 12 shipped). The four skipped were either premature optimization (UI granularity before profiling), speculative config (`openPositioning`), already-deleted-as-dead-code (worker thread), or infra-decision-blocked (grammar download mirroring). External AI audits are useful pre-filter material; ~25% noise rate is normal — don't take them as gospel, but don't dismiss them either.
- **TypeScript gotcha — `Action2.category`**: typed as `"File" | ILocalizedString | "View" | "Help" | "Test" | "Preferences" | "Developer" | undefined`. `localize()` returns `string` (the message text) — **not** assignable. `localize2()` returns `{ value, original }` matching `ILocalizedString`. For Action2 always use `localize2` for `title` and `category`. Pre-existing bug in `agentPanelActions.ts` that the toggled-state edit surfaced.
- **Multi-root path-collision strategy**: instead of inventing a prefix scheme (e.g. `<folder-name>/<rel>`) and migrating the DB schema, store chunks as `relative(workspaceRoot, abs)`. Files under sibling roots come out as `../folder2/src/foo.ts` — globally unique inside one DB, no schema change, single-root behaviour unchanged. Cheap and correct.
- **Incremental update semantics**: file-watcher pipeline does NOT remove stale manifest entries on file deletion (only `deleteByFile`). Rationale: the `walkWorkspace` walker only enumerates existing files, so a stale manifest entry referencing a gone path is never consulted — its mtime check is short-circuited by the file just not appearing in the next walk. Saves a round-trip + sidesteps the missing `deleteManifestEntry` API on the DB.
- **Suggested improvement for next turn**: the `retrieve()` 15-second timeout currently rejects the whole turn if the DB is slow. Consider a softer fallback — return cached/empty hits with a flag instead of throwing, so the agent can degrade rather than fail. Open question for Daniel before I commit to a behaviour change.

### 2026-05-27 — Briefing pre-compute cache shipped
- `runGetProjectBriefing` now consults a TTL-bounded cache on `LspBridgeAdapter` before fanning out to its 3 parallel IO ops (journal read, file-tree walk, `.git/logs/HEAD` parse) + optional `notes.listNotes()`. Cache key is `'with-notes' | 'no-notes'` (mirrors `params.includeNotes`). TTL is 30s — long enough to deduplicate within a single turn / multi-agent burst, short enough that an agent re-asking 60s later gets fresh data without anyone having to remember to invalidate.
- File-watch invalidation: global `IFileService.onDidFilesChange` subscription on the adapter clears the cache whenever any path ending in `/AGENTS.md`, `/.github/AGENTS.md`, `/.github/copilot-instructions.md`, or `/.git/logs/HEAD` is added/updated/deleted. Suffix-match is cheap (path.endsWith × 4 per changed file) and these events fire infrequently; cheaper than registering 4 path-scoped watchers each turn. The disposable is `_register`-ed so it lives with the singleton.
- Why not also pre-compute on workbench start? Considered registering an `IWorkbenchContribution` at `LifecyclePhase.Restored` that warms the cache, but it adds workbench-contribution wiring + a startup-time IO cost for users who never use CB. The cache + watcher already buys the win on the **first** call (sub-ms hit afterward for the rest of the session unless a journal file actually changes), which is what matters most for the agent-tool latency story. Skipped to keep this turn tight; revisit if telemetry shows first-call latency dominating.
- Interface contract: added `getCachedBriefing(key) / cacheBriefing(key, value) / invalidateBriefingCache()` to `ILspBridgeAdapter` so other tools or a future startup warmer can read/seed the cache without poking the implementation. ### 2026-05-27 — Stub-on-decl coverage lint shipped
- The B.1↔B.2 drift class is dead. `BuiltinToolResultType` is now the **single source of truth** — any key declared there MUST appear in `builtinTools` + `callTool` + `stringOfResult` or hygiene fails. Future B.3, Sprint 2, and Phase C tool additions cannot land half-wired without CI catching it.
- Design choice: pure-Node regex parser, no TS-compiler dependency. Brace-balanced block extractor honors strings (`'…' "…" `…``) and `// / /* */` comments so braces inside string literals or comments don't confuse depth tracking. Top-level key extraction works by masking everything at depth > 0 to spaces, then running a simple key regex against the masked text — handles all three property-name forms (`'foo':`, `"foo":`, `foo:`) without false positives from nested objects.
- Why not the TS compiler API? (a) hygiene runs early in the build, can't depend on a compile; (b) regex is fine for our control of these declaration sites; (c) the script doubles as a debugging tool you can run with no setup. If declaration shapes drift far enough that regex breaks, the script exits 2 with "parsed to zero keys" — fail-loud, not silent.
- Stale-key warning path is intentionally non-fatal. If someone removes a tool from `BuiltinToolResultType` but leaves it in the executor (during a deprecation), they get a warning telling them what to clean up, not a CI failure that blocks the deprecation PR.
- `process.execPath` used in the gulp task wrapper (not literal `node`) so it works under nvm-windows where `node` isn't on PATH for spawned subprocesses.

### 2026-05-27 — B.2 outline cache
- `LspBridgeAdapter.getDocumentSymbols` caches per `(uri, model.getVersionId())`. `versionId` is VS Code's canonical "did this model change" probe — monotonically increments on every edit, so cache invalidation is "free" (compare-and-skip, no listener). No `onModelRemoved` listener needed — the bounded LRU evicts stale entries naturally as new files are touched.

| `.ts` under `browser/` or `common/` (e.g. `toolsService.ts`, `prompts.ts`, `chatThreadService.ts`) | `.\dev.ps1 -FullGulp` (~3 min) | These compile via gulp into `out/` — fast path skips them |
| `.ts` under `electron-main/` (e.g. `webSearchChannel.ts`, `app.ts`) | `.\dev.ps1 -FullGulp` (~3 min) + **full V3Code restart** | Main process code loads ONCE at app launch — Ctrl+R does nothing |
| `void.css` (workbench CSS) | `.\dev.ps1 -Once` copies it | Fast path handles this |

**The #1 mistake**: editing a `.ts` service file, running `.\dev.ps1 -Once`, and wondering why nothing changed. The fast path ONLY rebuilds the React bundle. If you touched anything outside `react/src/`, you MUST use `-FullGulp`.

**The #2 mistake**: editing `electron-main/` code, running `-FullGulp`, and only doing Ctrl+R. The main process keeps the OLD code in memory. You must fully quit + relaunch V3Code.
